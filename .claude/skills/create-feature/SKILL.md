# Create Feature

## Purpose
Scaffold a new front-end feature screen (Screen, ViewModel, UIState, Event, Preview, ViewModelTest) using the devtools CLI, then wire up DI and navigation. Templates are embedded in the devtools JAR — no `.idea/fileTemplates` dependency.

## Prerequisites

Features live inside an **activity** nav graph. Before creating a feature, an activity must exist. If you haven't created one yet, run `/create-activity` first. Each activity has:
- `<Activity>ActivityScreen.kt` — the `NavGraphBuilder` extension
- `<Activity>Destination.kt` — the sealed class of typed destinations for that activity

## Step 1 — Gather information

Ask the user for:
- **Feature name** (PascalCase, e.g. `AddProperty`, `SelectOrg`)
- **Parent folder** — the **activity** directory one level below `features/` where the new feature package will be created.
  The feature package name is the feature name lowercased.

  For example, feature `AddProperty` with parent
  `edifikana/front-end/app/src/commonMain/kotlin/com/cramsan/edifikana/client/lib/features/auth`
  produces the package `com.cramsan.edifikana.client.lib.features.auth.addproperty`.

  ⚠️ The `--parent` path must point to the **activity** subdirectory, not to `features/` itself.
  Valid:   `.../features/auth`
  Invalid: `.../features`

## Step 2 — Run the CLI

```bash
./scripts/devtools create feature --name <FeatureName> --parent <activity-dir-path>
```

The script creates 6 files:
- `<Name>Screen.kt` — Composable screen wiring (commonMain)
- `<Name>Event.kt` — sealed class of ViewModelEvents (commonMain)
- `<Name>UIState.kt` — immutable UI state data class with `Initial` companion (commonMain)
- `<Name>ViewModel.kt` — extends `BaseViewModel` (commonMain)
- `<Name>ScreenPreview.kt` — `@Preview` composable (commonMain)
- `<Name>ViewModelTest.kt` — placed in the matching `jvmTest` source set

## Step 3 — Declare the navigation destination

The generated screen references `<Feature>Destination` but does **not** declare it. Open the parent
activity's `<Activity>Destination.kt`.

⚠️ If `<Activity>Destination.kt` still contains the generated `PlaceholderDestination` (and a
matching `startDestination = <Activity>Destination.PlaceholderDestination` in
`<Activity>ActivityScreen.kt`), **remove it** as part of this step instead of adding your new
destination alongside it: delete `PlaceholderDestination`, its `placeholderRoute` val, and its
entries in `fromWebPath`/`toWebPath`, then point `startDestination` at your new
`<Feature>Destination`.

Otherwise, add the new destination as a member of the sealed class:

```kotlin
@Serializable
data object <Feature>Destination : <Activity>Destination()
```

Then add a route val, a `toWebPath()` branch, and chain it onto `fromWebPath`/`toWebPath` —
follow the worked example in `<Activity>Destination.kt`'s class-level KDoc (generated by
`create activity`), which shows exactly this for a 2nd destination.

## Step 4 — Register the composable route

Open `<Activity>ActivityScreen.kt` and add a `composable` entry inside the `navigationGraph { }`
block, per its KDoc:

```kotlin
composable(<Activity>Destination.<Feature>Destination::class) {
    <Feature>Screen()
}
```

The KDoc on `<activity>NavGraphNavigation` (generated by `create activity`) calls out two gotchas
for this step: the missing `composable` import, and where `typeMap` is passed.

## Step 5 — Wire screen-to-screen navigation

If this feature navigates to another screen or back, follow the `emitWindowEvent` pattern
documented in `<Feature>ViewModel.kt`'s KDoc — do not call `navController.navigate(...)` directly,
and do not route navigation through `<Feature>Event`.

⚠️ Wiring navigation typically means adding a callback parameter to `<Feature>Content` (e.g.
`onDetailsClicked: () -> Unit`) so `<Feature>Screen` can pass `viewModel::navigateToDetails` in.
Whenever `<Feature>Content`'s parameters change, update the matching call in
`<Feature>ScreenPreview.kt` to pass the new callbacks (e.g. `onDetailsClicked = {}`) — the
generated preview only matches the initial parameter-less signature and will fail to compile
otherwise.

## Step 6 — Apply DI registration

The CLI prints a reminder like:
```
viewModelOf(::AddPropertyViewModel)
```
Read `ViewModelModule.kt` and insert that line inside the `scope<String> { ... }` block.

## Step 7 — Note on WASM / URL routing

No extra wiring needed: destinations declared on `<Activity>Destination` (a `WebDestination`) are
automatically routable from the browser address bar once Step 3 is done — `PathNavigation.kt` was
already wired to this activity's `toWebPath`/`fromWebPath` when it was created (see
`/create-activity`). A standalone top-level `Destination` (not part of a `WebDestination` sealed
class) would need manual wiring into `PathNavigation.kt`'s handler lists instead.

## Step 8 — Verify compilation

```bash
./gradlew :<app>:front-end:app:release --quiet
```
