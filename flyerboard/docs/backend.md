# FlyerBoard Back-End

## Overview

FlyerBoard is a web application for browsing and submitting event flyers. Visitors can browse the public feed without an account. Registered users can submit flyers for admin review. After an admin approves a submission it becomes publicly visible; once the event date passes the flyer moves automatically to an archive that is also publicly browsable.

The back-end is a **Ktor/Kotlin** server backed by **Supabase** (Postgres + Storage + Auth).

---

## Architecture

The server follows a strict three-layer architecture:

```
HTTP request
    │
    ▼
Controller          validates auth, extracts path/query/body params, maps to/from network models
    │
    ▼
Service             all business logic: input validation, ownership checks, orchestration
    │
    ▼
Datastore           data access only — Supabase Postgrest queries, file storage operations
```

Dependency injection is handled by **Koin**. Every layer depends only on interfaces, making each layer independently testable with MockK mocks.

### Module layout

| Location | Purpose |
|---|---|
| `flyerboard/api` | Shared API contract objects (`FlyerApi`, `ModerationApi`, `UserApi`) |
| `flyerboard/models` | Shared network request/response models used by client and server |
| `flyerboard/back-end/src/main/.../controller/` | Ktor route handlers |
| `flyerboard/back-end/src/main/.../service/` | Business logic services |
| `flyerboard/back-end/src/main/.../datastore/` | Datastore interfaces and Supabase implementations |
| `flyerboard/back-end/supabase/migrations/` | Postgres schema migrations |

## Authentication & Authorization

Authentication is delegated to **Supabase Auth**. Clients obtain a JWT from Supabase directly (email/password or OAuth) and pass it as `Authorization: Bearer <token>` on every protected request.

The `FlyerBoardContextRetriever` validates the token on each request by calling `auth.retrieveUser(token)` against Supabase, then looks up the caller's role from the `user_profiles` table. If no profile row exists for a valid auth user, one is created automatically with the `user` role.

### Roles

| Role | Capabilities |
|---|---|
| **Unauthenticated** | Browse public flyer feed and archive, view individual flyers |
| **user** | All of the above + submit flyers, update/delete own flyers, view own submissions, register display name |
| **admin** | All of the above + approve/reject any pending flyer (with optional reason), delete any flyer, change any user's role |

## Data Model

### `users` table
Stores the display name of registered users, keyed by their Supabase Auth UUID.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | References `auth.users(id)` |
| `first_name` | TEXT | |
| `last_name` | TEXT | |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | Auto-updated by trigger |

### `user_profiles` table
Stores the role of each authenticated user.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | References `auth.users(id)` |
| `role` | TEXT | `'user'` or `'admin'` |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | Auto-updated by trigger |

### `flyers` table

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | Auto-generated |
| `title` | TEXT | Max 200 chars (enforced in service) |
| `description` | TEXT | Max 2 000 chars (enforced in service) |
| `file_path` | TEXT | Path inside the `flyer-files` Storage bucket |
| `status` | TEXT | `pending` → `approved` / `rejected` → `archived` |
| `expires_at` | TIMESTAMPTZ | Optional event date; triggers archival when passed |
| `uploader_id` | UUID | References `user_profiles(id)` |
| `rejection_reason` | TEXT | Nullable; set by admin on rejection |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | Auto-updated by trigger |

### Flyer lifecycle

```
[submit]  PENDING  ──approve──▶  APPROVED  ──expires──▶  ARCHIVED
                   ──reject───▶  REJECTED
```

- Every new submission starts as **PENDING**.
- An admin can **approve** (makes it public) or **reject** (removes the file from Storage, records an optional reason).
- An hourly background job (`ExpiryService`) transitions approved flyers whose `expires_at` is in the past to **ARCHIVED**.
- Editing a flyer resets it to **PENDING** and triggers re-moderation.

### File storage

Files are stored in a private Supabase Storage bucket named `flyer-files`. Each file is stored with a UUID prefix to prevent collisions. Access is via short-lived signed URLs (1-hour expiry) generated by the service layer and attached to every flyer response.

Accepted MIME types: `image/jpeg`, `image/png`, `image/webp`, `application/pdf`. Maximum size: 10 MB (configurable via `flyer.max_file_size_bytes`).

## API Reference

Base URL: `/`

All JSON responses use `snake_case` field names. Timestamps are ISO-8601 strings.

### Authentication header

```
Authorization: Bearer <supabase-jwt>
```

Required on all endpoints marked **Auth required**. Admin-only endpoints additionally require the caller's `user_profiles.role` to be `admin`.

---

### Health

#### `GET /api/v1/health`
Returns server health status. Always public.

**Response 200**
```json
{ "message": "ok" }
```

---

### Users

#### `POST /user` — Register display name
**Auth required.** Creates a `users` row linking the caller's Supabase Auth identity to a display name. Call this once after the client-side Supabase Auth sign-up flow completes.

**Request body**
```json
{ "first_name": "Alice", "last_name": "Smith" }
```

**Response 200**
```json
{ "id": "<uuid>", "first_name": "Alice", "last_name": "Smith" }
```

---

### Flyers

#### `GET /api/v1/flyers` — List flyers
Public. Returns approved flyers by default. Supports optional status filter and full-text search.

**Query parameters**

| Parameter | Type | Default | Description |
|---|---|---|---|
| `offset` | int | 0 | Pagination offset |
| `limit` | int | 20 | Page size |
| `status` | string | — | Filter by status (`pending`, `approved`, `rejected`, `archived`) |
| `q` | string | — | Case-insensitive search against title and description |

**Response 200**
```json
{
  "flyers": [ { ...FlyerObject } ],
  "total": 42,
  "offset": 0,
  "limit": 20
}
```

---

#### `GET /api/v1/flyers/{id}` — Get flyer
Public. Returns a single flyer by ID.

**Response 200** — `FlyerObject`
**Response 404** — flyer not found

---

#### `GET /api/v1/flyers/archive` — Browse archive
Public. Returns archived flyers with optional search. Supports the same query parameters as the main listing (`offset`, `limit`, `q`).

**Response 200** — same shape as flyer list

---

#### `GET /api/v1/flyers/mine` — My submissions
**Auth required.** Returns all flyers uploaded by the authenticated user across all statuses.

**Query parameters:** `offset`, `limit`

**Response 200** — same shape as flyer list
**Response 401** — not authenticated

---

#### `POST /api/v1/flyers` — Submit a flyer
**Auth required.** Creates a new flyer with `pending` status. Body must be `multipart/form-data`.

**Form fields**

| Field | Type | Required | Description |
|---|---|---|---|
| `title` | string | Yes | Flyer title (stripped of HTML, max 200 chars) |
| `description` | string | Yes | Description (stripped of HTML, max 2 000 chars) |
| `expires_at` | ISO-8601 string | No | Event date; flyer is archived after this time |
| `file` | binary | Yes | Image or PDF (JPEG, PNG, WebP, PDF; max 10 MB) |

**Response 200** — `FlyerObject` with `status: "pending"` and a signed `file_url`
**Response 400** — unsupported file type or file too large
**Response 401** — not authenticated

---

#### `PUT /api/v1/flyers/{id}` — Update a flyer
**Auth required.** Updates a flyer. Only the original uploader may call this. Resets status to `pending` and triggers re-moderation. Body must be `multipart/form-data`.

**Form fields** — all optional; omitting a field leaves it unchanged.

| Field | Description |
|---|---|
| `title` | New title |
| `description` | New description |
| `expires_at` | New event date |
| `file` | Replacement file (same constraints as on create) |

**Response 200** — updated `FlyerObject` with `status: "pending"`
**Response 401** — not authenticated
**Response 403** — caller is not the uploader
**Response 404** — flyer not found

---

#### `DELETE /api/v1/flyers/{id}` — Delete a flyer
**Auth required.** Deletes the flyer record and its associated file from Storage. The uploader or any admin may call this.

**Response 204** — deleted successfully
**Response 401** — not authenticated
**Response 403** — caller is not the uploader and is not an admin
**Response 404** — flyer not found

---

### Moderation (admin only)

All moderation endpoints require `role: admin`. A `403` is returned if the caller is authenticated but not an admin; `401` if unauthenticated.

#### `GET /api/v1/moderation` — List pending flyers
Returns all flyers with `pending` status, paginated.

**Query parameters:** `offset`, `limit`

**Response 200** — flyer list (same shape as public listing)

---

#### `POST /api/v1/moderation/{id}` — Approve or reject a flyer

**Request body**
```json
{ "action": "approve" }
```
```json
{ "action": "reject", "reason": "Does not meet community guidelines" }
```

| Field | Values | Description |
|---|---|---|
| `action` | `"approve"` \| `"reject"` | The moderation decision |
| `reason` | string (optional) | Rejection reason stored on the flyer and returned to the submitter |

**Approve** — transitions flyer to `approved`; file is kept.
**Reject** — transitions flyer to `rejected`; file is deleted from Storage.

**Response 200** — updated `FlyerObject`
**Response 400** — invalid action value
**Response 404** — flyer not found

---

#### `PATCH /api/v1/moderation/users/{id}/role` — Change a user's role

**Request body**
```json
{ "role": "admin" }
```

| Field | Values |
|---|---|
| `role` | `"user"` \| `"admin"` |

**Response 200**
```json
{ "id": "<uuid>", "role": "admin" }
```
**Response 400** — unrecognised role value

---

### `FlyerObject` shape

```json
{
  "id": "3fa85f64-...",
  "title": "Summer Concert",
  "description": "Live music at the park",
  "file_url": "https://...",
  "status": "approved",
  "expires_at": "2026-07-04T20:00:00Z",
  "uploader_id": "9cf1b7...",
  "rejection_reason": null,
  "created_at": "2026-05-01T10:00:00Z",
  "updated_at": "2026-05-01T12:00:00Z"
}
```

`file_url` is a short-lived signed URL (1-hour TTL). `rejection_reason` is non-null only on flyers with `status: "rejected"`.

## Configuration

Settings are loaded from `config.properties` with environment variable overrides. The environment variable name is `<DOMAIN>_<KEY>` with dots replaced by underscores and uppercased (e.g. `FLYERBOARD_SUPABASE_URL`).

| Key | Default | Description |
|---|---|---|
| `supabase.url` | `http://127.0.0.1:54321` | Supabase project URL |
| `supabase.key` | *(test key)* | Supabase service role key |
| `flyer.max_file_size_bytes` | `10485760` (10 MB) | Maximum upload size in bytes |
| `logging.account_created` | — | Log a line each time a user account is created |

## Testing

### Unit tests (`src/test/`)

Service tests use MockK to mock the datastore layer and verify business logic in isolation. Controller tests use Ktor's `testBackEndApplication` DSL with mocked services to verify HTTP semantics (status codes, auth enforcement, response shape).

```bash
./gradlew :flyerboard:back-end:test
```

### Integration tests (`src/integTest/`)

**Controller integration tests** wire the real controllers against mocked services using Ktor's test engine — they verify routing, auth enforcement, and serialization end-to-end without touching a database.

**Datastore integration tests** run against a real Supabase instance (configured in `config.properties.integ`) and verify actual SQL queries, constraints, and data integrity.

```bash
# Controller integration tests (no database required)
./gradlew :flyerboard:back-end:integTest

# Datastore integration tests (requires a running Supabase instance)
./gradlew :flyerboard:back-end:integTest --tests "*DatastoreIntegrationTest*"
```
