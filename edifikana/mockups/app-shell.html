App Shell

The structural shell shared by every authenticated screen in the Edifikana app. This document covers the DOM structure, each layer's role, breakpoint behavior, active-state rules, shell type classification, and common mistakes to avoid.

Every authenticated screen nests seven structural layers. The preview below forces all layers visible simultaneously — in production the sidebar is hidden on mobile/tablet and the bottom nav is hidden on desktop.

.mobile-viewport
Outermost container. Controls the visible frame width: 375px on mobile (phone frame shadow), 768px on tablet (no shadow), 100% on desktop. On desktop the background switches to bg-warm to show page background around the constrained content column.
.app-layout
Primary layout engine. flex-direction: column on mobile/tablet — the sidebar is hidden so this is invisible. Switches to flex-direction: row on desktop, placing the sidebar to the left. Must wrap every authenticated screen — without it the sidebar never appears on desktop.
.screen
The main content column. Takes flex: 1 inside .app-layout. On desktop, constrained to max-width: 900px (--content-max-width) so content doesn't stretch to full page width.
.sidebar-nav
display: none on mobile and tablet. Appears on desktop (≥1024px) as a 240px wide sticky column with height: 100vh and white background. Use .sidebar-nav__item--active on the item matching the current section.
.top-bar
Always visible. position: sticky; top: 0; z-index: 100. Minimum height 56px. Two patterns: .top-bar__title for primary tab screens, .top-bar__back for detail and form screens. Gains a subtle bottom border on desktop.
.screen-content
The scrollable body. flex: 1; overflow-y: auto. Has padding-bottom: 80px on mobile/tablet to clear the fixed bottom nav. On desktop drops to 24px. Use .screen-content--no-nav when the screen deliberately omits the bottom nav — this removes the 80px gap.
.fab
Optional floating action button. position: fixed; z-index: 150. On mobile sits 80px above the bottom nav. On desktop repositions to right: 32px; bottom: 32px. Omit entirely when the screen has no primary creation action.
.bottom-nav
Mobile/tablet tab bar. position: fixed; bottom: 0; z-index: 200; height: 64px. Respects env(safe-area-inset-bottom) for notched devices. Hidden on desktop — the sidebar replaces it. Use .bottom-nav__item--active on the same section as .sidebar-nav__item--active.

Active State Rules

The active nav item must be marked in both the sidebar and the bottom nav — they are two surfaces for the same navigation state. A mismatch causes different tabs to highlight depending on the user's screen size.

⚠️
Rule: Add .sidebar-nav__item--active and .bottom-nav__item--active to the same destination href in each nav block. If the current screen lives under Tasks, both the sidebar Tasks link and the bottom nav Tasks item must carry their respective active modifier — never just one.

Task List, Task Detail, Add/Edit Task, Request Review task-list.html task-list.html
Settings, Profile, Edit Profile, Change Password, Team Management, Notifications, Documents, My Organizations, Organization Detail, Create Organization, Join Organization, Delete Account settings.html settings.html

Top Bar Variants

Choose the top-bar pattern based on the screen's position in the navigation hierarchy.

Pattern When to use HTML
Title bar Primary nav destinations: Dashboard, Property List, Task List, Settings (the four tab screens) <span class="top-bar__title">Title</span>
Back bar Detail screens, form screens, and settings sub-screens that sit below a primary tab <a href="parent.html" class="top-bar__back">← Label</a>
Icon actions Any bar — add trailing icon buttons alongside either pattern above <div class="top-bar__actions"><button class="top-bar__icon-btn">…</button></div>

Screen Type Classification

Not all screens use the full App Shell. Choose the correct shell for the screen's context before writing any markup.

Type 1 — Most screens

App Shell

All authenticated screens within the main navigation. Use the full template: .app-layout + .sidebar-nav + .bottom-nav. Mark the active section in both nav elements.

  • Dashboard
  • Property List / Detail
  • Unit List / Detail, Common Area
  • Task List / Detail, Request Review
  • Settings and all sub-screens
  • My Organizations, Org Detail
  • Profile, Team Management, Notifications
Type 2 — Pre-auth

Auth (No Shell)

Screens shown before or outside the main navigation. No sidebar, no bottom nav. Use .auth-layout inside .mobile-viewport > .screen to center content vertically.

  • Sign In
  • Sign Up
  • Password Reset
  • OTP Validation
  • Set New Password
  • Invitation Accept
Type 3 — Post-auth setup

Setup Flow (Partial)

Post-auth screens before the user has an active org context. Use .mobile-viewport > .screen with no .app-layout. Use .screen-content--no-nav and .top-bar__back.

  • Organization Picker (post-login, multi-org)
  • Create Organization (first time, no org yet)
ℹ️
Organization screens reached from Settings are App Shell screens. Create Organization, Join Organization, and Organization Detail accessed via Settings → My Organizations use the full .app-layout shell with "More" as the active tab — not the Setup Flow partial shell.

Common Mistakes

Mistake Symptom Fix
Missing .app-layout No sidebar on desktop; content stretches full width Wrap .sidebar-nav + .screen in <div class="app-layout">
Missing .sidebar-nav Sidebar column absent even when .app-layout is present Add <nav class="sidebar-nav"> as the first child of .app-layout
Mismatched active states Different tab highlighted on mobile vs desktop Set --active on the same destination in both nav elements
.screen-content without --no-nav on a no-nav screen 80px empty gap at the bottom on mobile Use .screen-content--no-nav when the screen has no .bottom-nav
.screen-content--no-nav on a primary tab screen Content overlaps the fixed bottom nav on mobile Use plain .screen-content on all four primary tab screens
Hardcoded hex colours or raw pixel values Mismatches design tokens; breaks future theming Use var(--color-*), var(--radius-*), var(--space-*), var(--shadow-*)
FAB on a read-only or list-only screen with no creation action Confusing affordance — user expects a create action Omit .fab entirely; only include it when there is a primary creation action
Using .auth-layout structure for an authenticated screen No sidebar, no bottom nav on a screen that needs them Use the full App Shell template with .app-layout

New Screen Checklist

Self-verify each item before marking a screen mockup complete.

  • Correct shell type applied — App Shell, Auth, or Setup Flow
  • .app-layout wraps .sidebar-nav + .screen (App Shell only)
  • .sidebar-nav present with the correct --active item (App Shell only)
  • .bottom-nav present with the correct --active item (App Shell only)
  • Sidebar and bottom-nav active items point to the same destination href
  • .screen-content vs .screen-content--no-nav chosen correctly
  • Top-bar pattern is correct — .top-bar__title for primary tabs, .top-bar__back for detail and form screens
  • No hardcoded hex colours — all colour values use var(--color-*) tokens
  • No raw pixel values for spacing, radius, or shadow — use var(--space-*), var(--radius-*), var(--shadow-*)
  • All href values point to real .html files, or # with an explanatory HTML comment
  • FAB included only when the screen has a clear primary creation action
  • All distinct use-cases modelled — empty state, happy path, and error state as applicable
